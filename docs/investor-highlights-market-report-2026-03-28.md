# 투자자용 특장점 및 시장성 보고서

기준일: `2026-03-28`

프로젝트명: `Threads Archive`

## 1. 한 줄 요약

`Threads Archive`는 빠르게 성장 중인 Threads 생태계에서, 흘러가는 대화를 `저장 가능한 지식 자산`, `개인 아카이브`, `재사용 가능한 문서`로 전환하는 `capture layer` 제품이다.

핵심 포지션은 단순 클리퍼가 아니라 다음과 같다.

- `Threads 특화 저장 인프라`
- `개인 지식화 워크플로우`
- `로컬 저장 + SaaS scrapbook + 향후 팀/리서치 확장 가능한 데이터 레이어`

## 2. 왜 지금인가

Threads는 더 이상 “출시 초기의 보조 소셜 앱”이 아니다.

공개 자료 기준으로:

- Meta는 `2024년 7월` Threads가 `175M monthly actives`를 넘었다고 공식 언급했다.
- 이후 Threads는 `web experience`, `custom feeds`, `tagged topics`, `messaging`, `communities`, `text attachments`, `fediverse` 연동 확대 등 제품 깊이를 지속적으로 확장했다.
- Semrush 기준 `threads.com`은 `2026년 2월`에 `187.56M` visits를 기록했고, 핵심 웹 트래픽 국가는 `Taiwan`, `United States`, `Japan`, `India`, `Brazil` 순으로 나타난다.

이 말은 곧, Threads가 `짧은 반응형 피드`에서 점점 `의견`, `전문성`, `커뮤니티`, `리서치 가치가 있는 대화`가 쌓이는 장소로 이동하고 있다는 뜻이다.

플랫폼이 깊어질수록 생기는 문제는 명확하다.

- 좋은 글이 금방 묻힌다
- 나중에 다시 찾기 어렵다
- 답글 체인 문맥이 깨진다
- 북마크는 저장이지 정리가 아니다
- 팀/개인 문서 도구로 연결되지 않는다

Threads Archive는 바로 이 지점에서 가치가 생긴다.

## 3. 제품 특장점

## 3-1. Threads 전용 문제를 정확히 푸는 제품

대부분의 저장 도구는 범용 웹 클리퍼다.

하지만 Threads는 범용 웹페이지와 다르다.

- 본문 길이가 짧다
- 작성자 자기 답글이 본문 의미를 완성한다
- 미디어와 링크가 섞인다
- 저장 시점에 구조화되지 않으면 재사용 가치가 급감한다

Threads Archive의 강점은 `Threads 문법`을 알고 있다는 점이다.

핵심 차별점:

- 개별 포스트 추출
- 작성자 자기 답글 체인 통합
- 이미지/미디어 처리
- Markdown / ZIP / 클라우드 아카이브 저장
- 향후 Notion/문서 도구 연동 확장

즉, “웹페이지 저장”이 아니라 `Threads conversation unit 저장`이다.

## 3-2. 로컬-퍼스트와 SaaS를 모두 잡는 하이브리드 구조

제품은 두 가지 가치를 동시에 제공한다.

- 로컬 저장: Obsidian, ZIP, 개인 소유 데이터
- SaaS 저장: Threads Archive Cloud scrapbook

이 구조의 장점:

- power user는 자기 워크플로우를 유지할 수 있다
- 일반 사용자는 별도 폴더/파일 관리 없이 클라우드에서 바로 쓸 수 있다
- 장기적으로 개인용 도구에서 협업/검색/리포트 제품으로 확장할 수 있다

이 하이브리드 구조는 투자 관점에서 중요하다.

- 진입은 가볍다
- retention 포인트는 많다
- 로컬 제품에서 SaaS로 업셀 가능하다

## 3-3. 제품 사용 자체가 유입이 되는 구조

이 제품의 가장 큰 장점 중 하나는 `사용 행위가 곧 마케팅`이라는 점이다.

Mention scrapbook 플로우에서는 사용자가 Threads 답글에 봇 멘션을 단다.  
이 멘션은 다른 사용자에게 그대로 노출되고, 그 과정에서 봇 프로필과 scrapbook 제품으로 자연스럽게 유입이 발생한다.

즉 구조가 다음처럼 작동한다.

- 사용자가 봇을 멘션한다
- 그 멘션이 다른 사용자에게 보인다
- 다른 사용자가 봇 프로필을 방문한다
- 프로필에서 제품 가치와 사용 방식을 이해한다
- 추가 사용자가 다시 봇 멘션을 남긴다

이건 투자자 관점에서 매우 중요한 특성이다.

- 별도 paid marketing 의존도가 낮다
- product usage와 user acquisition이 분리되지 않는다
- Threads 내부에서 자연스러운 distribution loop가 생긴다
- creator / researcher 커뮤니티에서 반복 노출이 누적된다

즉, 일반적인 툴 SaaS처럼 CAC를 매번 돈으로 사는 구조가 아니라, `제품 사용이 곧 공개적 데모`가 되는 구조다.

## 3-4. AI를 원가 부담이 작은 형태로 설계

제품의 AI는 core feature가 아니라 `정리 강화 레이어`로 설계되어 있다.

현재 방향:

- BYO API Key
- 요약, 태깅, frontmatter 정리
- 저장 후 문서 정리 품질 강화

장점:

- inference cost를 회사가 전액 떠안지 않는다
- gross margin을 방어하기 쉽다
- AI 품질 경쟁보다 `workflow integration`에 집중할 수 있다

투자자 관점에서는 “AI가 있느냐”보다 “AI가 unit economics를 망치지 않느냐”가 더 중요하다. 이 구조는 그 점에서 유리하다.

## 3-5. 다국어 및 글로벌 확장 준비가 이미 되어 있음

현재 지원 언어:

- `ko`
- `en`
- `ja`
- `pt-BR`
- `es`
- `zh-TW`
- `vi`

이건 단순 번역이 아니라, Threads 트래픽이 강한 국가군에 맞춘 우선순위다.

즉 제품은 이미 `국내 유틸`이 아니라 `초기 글로벌 시장 대응형` 구조를 갖췄다.

## 3-6. 운영 콘솔과 상업 운영 하드닝까지 진입

대부분 초기 툴 제품은 기능만 만들고 운영 레이어가 약하다.

Threads Archive는 이미 다음을 갖췄다.

- admin console
- 결제/주문/라이선스/이메일 운영
- cloud save 운영 API
- runtime config 관리
- DB / SMTP 테스트
- 보안 하드닝
- 상업 운영 체크리스트 및 운영 문서

이건 단순 MVP를 넘어 `실제로 팔고 운영하는 제품` 쪽에 더 가깝다.

## 4. 시장성

## 4-1. 시장 정의

이 제품의 직접 시장은 다음 3개가 겹치는 영역이다.

- `Threads heavy users`
- `지식 저장 / research / creator workflow users`
- `문서 도구 사용자`

즉, 대중형 소셜 유저 전체를 노리는 것이 아니라 `가치 밀도가 높은 저장 수요층`을 노린다.

이 포지션은 장점이 있다.

- 대형 플랫폼 정면 승부가 아니다
- 사용자가 도구 구매에 익숙하다
- 저장 실패/수동 복사의 불편이 반복되는 고통점이다
- 멘션 기반 공개 노출 덕분에 낮은 CAC로 커뮤니티 침투가 가능하다

## 4-2. Threads 자체 시장 신호

공개 자료 기준 Threads는 국가별로 꽤 뚜렷한 강세 시장이 있다.

DataReportal의 `late 2025` 기준 Meta ad audience 추정:

- 미국: `24.6M`
- 일본: `13.0M`
- 멕시코: `7.40M`
- 대만: `6.65M`
- 베트남: `4.65M`
- 인도: `26.6M`

이 중 현재 제품이 직접 언어 대응하는 핵심 시장만 합쳐도, 공개 추정치 기준 최소 `56M+` 수준의 가시 사용자 풀을 설명할 수 있다.

여기에 인도의 영어 사용층, 브라질의 포르투갈어 시장, 대만/홍콩의 번체중국어 사용자층이 더해진다.

주의:

- 위 수치는 Meta 광고 도달치 기반 추정이므로 MAU와 정확히 동일하지 않다
- 하지만 투자 판단용 시장 signal로는 충분히 유효하다

## 4-3. 웹 트래픽 기준 시장 강도

Semrush 기준 `threads.com`의 `2026년 2월` 웹 트래픽 상위 국가는:

- 대만 `20.61%`
- 미국 `16.82%`
- 일본 `8.76%`
- 인도 `5.58%`
- 브라질 `4.49%`

또한 `threads.net`의 `2026년 2월` 상위 국가는:

- 미국 `22.63%`
- 대만 `13.41%`
- 베트남 `7.74%`

이 데이터는 매우 중요하다.

왜냐하면 Threads Archive는 웹/브라우저 확장 기반 제품이기 때문이다.

즉, `앱 사용자 수`보다 `웹 사용 강도`가 실제 제품 적합도에 더 가까운 선행 지표다.

Threads Archive가 선택한 언어와 시장 우선순위는 이 흐름과 잘 맞는다.

## 4-4. 왜 시장이 커질수록 우리 제품 가치가 커지는가

Threads는 공식적으로 다음 방향으로 제품을 확장하고 있다.

- custom feeds
- topic tags
- communities
- messaging
- longer-form text attachments
- web experience 강화
- fediverse 연동

이 변화는 모두 같은 결과를 만든다.

- 더 많은 전문 대화
- 더 긴 맥락
- 더 강한 커뮤니티
- 더 많은 “다시 보고 싶은” 글

즉, Threads가 커질수록 archive/capture 제품의 필요성도 같이 커진다.

우리는 Threads의 성장에 역행하는 제품이 아니라, Threads의 성숙과 함께 같이 커지는 제품이다.

## 4-5. TAM / SAM / SOM 및 예상 수익 시뮬레이션

이 섹션은 `투자자 설명용 보수적 모델`이다.

- `TAM`은 공개된 국가별 Threads 사용자 추정치를 바탕으로 계산했다.
- `SAM`, `SOM`, 매출 예측은 제품 적합도와 가격 정책을 반영한 `가정 기반 추론`이다.
- 기준 가격은 현재 랜딩 페이지의 공개 가격인 `one-time $29`를 사용했다.
- cloud add-on은 아직 공식 공개 가격이 없으므로, 아래 SaaS 시뮬레이션은 `가정값 $6 / month`를 둔 모델이다.

중요한 점:

- DataReportal의 Threads 수치는 `late 2025 Meta ad reach` 기반이며, 엄밀한 MAU와 동일하지 않다.
- 따라서 아래 수치는 `정밀 실사 수치`가 아니라 `투자 판단용 보수적 sizing`으로 봐야 한다.

### 4-5-1. Conservative TAM

현재 지원 언어와 직접 맞물리는 국가/지역 중, 이번 보고서에서 공개 수치를 직접 다시 검증한 시장만 합산하면 다음과 같다.

| 언어권 | 포함 국가/지역 | Threads 사용자 추정치 |
|---|---|---:|
| 영어 | 미국 `24.6M` + 인도 `26.6M` + 캐나다 `2.30M` + 호주 `1.35M` | `54.85M` |
| 일본어 | 일본 `13.0M` | `13.00M` |
| 스페인어 | 멕시코 `7.40M` + 아르헨티나 `3.35M` + 콜롬비아 `2.70M` | `13.45M` |
| 번체중국어 | 대만 `6.65M` + 홍콩 `1.55M` | `8.20M` |
| 베트남어 | 베트남 `4.65M` | `4.65M` |
| 합계 | 보수적 합산 TAM | `94.15M` |

이 `94.15M`은 의도적으로 보수적이다.

- `pt-BR`의 핵심 시장인 브라질은 Semrush 기준 Threads 웹 트래픽 상위 국가지만, 이번 합산에서는 제외했다.
- 영어권에서도 영국, 뉴질랜드 등 일부 시장을 제외했다.

따라서 현재 지원 언어 기준의 확장형 TAM은 실제로는 `100M+`로 보는 편이 더 자연스럽다.  
이 `100M+` 평가는 위의 직접 합산 수치와, 브라질/영국의 시장 강도를 함께 고려한 `추론`이다.

### 4-5-2. SAM 가정

이 제품의 실제 서비스 가능 시장은 Threads 전체가 아니라 다음 사용자군이다.

- 저장 욕구가 반복적으로 발생하는 creator
- 리서치 / 문서화 / PKM 습관이 있는 knowledge worker
- 브랜드, 커뮤니티, 인사이트를 장기 추적하는 운영자

그래서 보수적으로 보면, `지원 언어권 Threads 사용자 중 2%~5%` 정도가 실제 서비스 가능 시장이라고 보는 것이 합리적이다.

| 구분 | 가정 | 계산 결과 |
|---|---|---:|
| Low SAM | TAM의 `2%` | `1.88M` |
| Base SAM | TAM의 `3%` | `2.82M` |
| High SAM | TAM의 `5%` | `4.71M` |

이 중 투자자 설명용 기본값은 `Base SAM 2.82M`으로 두는 것이 적절하다.

### 4-5-3. SOM 가정

3년 내 현실적인 획득 가능 시장은 `Base SAM 2.82M` 중 일부다.

제품 특성상 대중 광고보다 다음 채널이 잘 맞는다.

- Threads creator / researcher 커뮤니티
- Obsidian / PKM / Notion 사용자층
- product-led 유입과 콘텐츠 기반 확산

이 구조를 고려하면, 3년 내 SOM은 다음 정도가 방어 가능하다.

| 시나리오 | Base SAM 대비 점유 | 3년 누적 유료 사용자 |
|---|---|---:|
| Conservative | `1.0%` | `28K` |
| Base | `2.5%` | `70K` |
| Aggressive | `5.0%` | `141K` |

여기서 중요한 점은, `Base 시나리오 70K paid`도 전체 보수적 TAM `94.15M`의 약 `0.07%` 수준이라는 것이다.  
즉, 매우 작은 침투율만으로도 의미 있는 사업 규모를 만들 수 있다.

### 4-5-4. Core 매출 시뮬레이션

현재 공개 가격 `one-time $29` 기준의 단순 매출 모델은 다음과 같다.

| 시나리오 | 3년 누적 유료 사용자 | one-time 가격 | 3년 누적 core bookings |
|---|---|---:|---:|
| Conservative | `28K` | `$29` | `$812K` |
| Base | `70K` | `$29` | `$2.03M` |
| Aggressive | `141K` | `$29` | `$4.09M` |

이건 recurring revenue가 아니라 `누적 일회성 판매 기준 gross bookings`다.

하지만 장점은 명확하다.

- 초기에는 low-friction 결제로 빠르게 PMF를 검증할 수 있다
- gross margin이 높다
- cloud / team / monitoring add-on을 붙이면 recurring revenue layer를 나중에 얹을 수 있다

### 4-5-5. Optional Cloud ARR 시뮬레이션

아직 공식 가격은 아니지만, 투자자 관점의 구조 이해를 위해 `cloud save / cloud archive`를 `월 $6` add-on으로 가정하면 다음과 같은 SaaS 그림이 나온다.

가정:

- Conservative: 유료 사용자 중 `15%` attach
- Base: 유료 사용자 중 `25%` attach
- Aggressive: 유료 사용자 중 `35%` attach

| 시나리오 | 3년 누적 유료 사용자 | cloud attach | 가정 cloud subscribers | 가정 cloud ARR |
|---|---|---:|---:|---:|
| Conservative | `28K` | `15%` | `4.2K` | `$302K ARR` |
| Base | `70K` | `25%` | `17.5K` | `$1.26M ARR` |
| Aggressive | `141K` | `35%` | `49.4K` | `$3.55M ARR` |

주의:

- 위 ARR은 `cloud SKU가 출시되고, 해당 attach rate가 성립한다고 가정한 run-rate`다.
- 현재 공개 가격은 `one-time $29`이며, cloud는 아직 실운영 가격이 정해진 상태가 아니다.

### 4-5-6. 투자자에게 직관적인 민감도 한 줄

보수적 TAM `94.15M`에서 단 `0.05%`만 유료 전환돼도:

- 유료 사용자 `47K`
- core bookings 약 `$1.37M`

`0.10%` 유료 전환이면:

- 유료 사용자 `94K`
- core bookings 약 `$2.73M`

즉, 이 사업은 “엄청난 플랫폼 장악”이 아니라 `매우 작은 침투율`만으로도 의미 있는 매출을 만들 수 있는 구조다.

## 5. 투자 포인트

## 5-1. 작은 팀으로 시작 가능한 고마진 도구 비즈니스

이 제품은 다음 특성을 가진다.

- 브라우저 확장 + 웹 SaaS 조합
- 고가 인프라가 필수는 아님
- BYO AI 구조로 비용 방어 가능
- 콘텐츠 생성보다 저장/정리 레이어라 moderation 비용 상대적으로 낮음

즉, 초기에는 lean하게 운영하면서도 마진 구조를 방어하기 좋다.

## 5-2. 특정 플랫폼 특화로 PMF 탐색이 빠름

너무 넓은 “모든 소셜 저장”이 아니라 `Threads` 하나에 집중하기 때문에:

- 메시지가 명확하다
- 사용 사례가 선명하다
- 커뮤니티 침투가 쉽다
- 제품 피드백 루프가 짧다
- 사용 자체가 공개 노출을 만들기 때문에 acquisition loop가 내장된다

초기 투자 관점에서는 이것이 오히려 장점이다.

작은 제품이지만 “누구를 위한 제품인지”가 명확하다.

## 5-3. 장기 확장성이 있다

현재는 Threads 저장 도구지만, 장기적으로는 다음 레이어로 확장 가능하다.

- creator archive
- personal knowledge capture
- research capture
- brand / mention archive
- team scrapbook
- intelligence / monitoring workflow

즉, 시작은 niche utility지만, 데이터 구조와 저장 레이어를 잡으면 `workflow platform`으로 커질 수 있다.

## 5-4. 경쟁이 약한 영역

현재 경쟁 대안은 대체로 다음 세 종류다.

- 수동 복사 / 북마크 / 스크린샷
- 범용 웹 클리퍼
- 일부 소셜 관리 도구

하지만 이들은 `Threads native extraction quality + reply chain preservation + local/SaaS hybrid` 조합을 거의 제공하지 않는다.

즉, 큰 시장에서 대형 경쟁사가 장악한 정면 승부가 아니라 `제품 공백이 있는 틈새`다.

## 6. 수익화 논리

현재 및 확장 가능한 수익화 구조:

- Pro 1회 결제
- update renewal
- cloud save / storage tier
- advanced organization
- team/shared archive
- creator/brand monitoring add-on

이 구조는 좋다.

- B2C로 시작 가능
- 이후 prosumer/team으로 확장 가능
- low-friction entry, higher-ARPU expansion이 가능

## 7. 왜 이 팀이 할 수 있는가

현재 코드베이스 기준으로 이미 갖춘 것:

- 실제 동작하는 extension
- local save
- cloud save
- scrapbook web app
- admin console
- 다국어 인프라
- 보안/운영 하드닝
- 패키지 구조 정리

즉, 아직 “아이디어 단계”가 아니다.

이미:

- 제품이 존재하고
- 운영 도구가 있고
- 상업용으로 닫아야 할 이슈를 식별했고
- 그 상당 부분을 해결했다

이건 execution risk를 낮춘다.

## 8. 리스크와 대응

## 8-1. 플랫폼 의존 리스크

리스크:

- Threads UI/API 변화
- Meta 정책 변화

대응:

- Threads 특화 extractor 유지
- web capture와 structured extraction 병행
- local-first / SaaS hybrid로 사용처 다변화

## 8-2. niche market 리스크

리스크:

- 너무 작은 시장으로 보일 수 있음

대응:

- Threads only utility에서 시작하되
- personal knowledge / creator / research / monitoring 레이어로 확장
- Notion, cloud archive, team use case로 TAM 확장

## 8-3. 상업 운영 리스크

리스크:

- 초기 운영에서 보안/DB/관리 콘솔 이슈 발생 가능

대응:

- admin 보안 하드닝 적용
- runtime config 분리
- Postgres 전환 경로 준비
- 운영 주의사항 문서화

## 9. 현재 투자자에게 보여줄 수 있는 메시지

가장 설득력 있는 메시지는 다음이다.

`Threads는 이미 충분히 큰 글로벌 플랫폼이 되었고, 점점 더 깊은 대화와 커뮤니티 중심의 공간으로 진화하고 있다. Threads Archive는 이 흐름에서 생기는 저장·재사용·문서화 문제를 푸는 가장 빠른 capture layer다. 특히 사용자가 봇을 멘션하는 행위 자체가 공개 노출과 프로필 유입을 만들어 별도 마케팅 비용 없이 distribution loop를 형성한다. 이미 제품, 운영 콘솔, 클라우드 저장, 다국어, 보안 하드닝까지 구현되어 있어 아이디어가 아니라 실행 단계의 자산이다.`

## 10. 결론

Threads Archive는 다음 특성을 가진 투자 검토 대상이다.

- 작은 팀으로 시작 가능한 고마진 생산성 도구
- Threads 성장의 수혜를 받는 구조
- 글로벌 언어/시장 확장 준비 완료
- local utility에서 SaaS/workflow platform으로 확장 가능
- 제품과 운영 레이어가 이미 상당 부분 구현됨

따라서 투자 관점에서의 현재 평가는 다음과 같다.

- 제품 선명도: `높음`
- 시장 타이밍: `좋음`
- 초기 글로벌 확장성: `높음`
- 수익화 설계: `유리`
- 운영 준비도: `상당히 진척`
- 장기 플랫폼 확장성: `높음`

최종 한 줄은 이렇다.

`Threads Archive는 Threads라는 성장 플랫폼 위에서, 저장되지 않는 대화를 자산으로 바꾸는 인프라형 도구다.`
