# 로그인 세션 안정성 유지 방안

기준일: `2026-03-30`

## 목표

로그인 세션은 다음 조건에서 끊기지 않는 것을 목표로 한다.

- 정상적인 배포와 서버 재시작 후에도 유지된다.
- 사용자가 서비스에 실제로 접속 중인 동안은 계속 연장된다.
- 장시간 비접속에 따른 만료는 정상 동작으로 간주한다.
- 운영 장애는 세션 자체보다 저장소, 프록시, 인증 설정에서 먼저 탐지한다.

## 현재 코드 기준

- 운영 환경은 파일 저장소가 아니라 Postgres를 강제한다.
- `threads_bot_session` 쿠키는 `HttpOnly`, `SameSite=Lax`, `Secure` 조건부 발급이다.
- 세션은 `GET /api/public/bot/session` 또는 `GET /api/public/bot/plus` 같은 접근 시 갱신된다.
- scrapbook 화면은 시작 시점에 `/api/public/bot/session`을 호출한다.
- 세션 수명은 30일이고, 마지막 활동 시각과 만료 시각이 함께 연장된다.

## 유지 원칙

1. 운영 저장소는 Postgres 단일 원천으로 고정한다.
2. 세션 인증은 쿠키 하나에 의존하되, 쿠키 정책은 변경하지 않는다.
3. 세션 갱신은 로그인 성공 이후의 실제 사용자 활동에만 연결한다.
4. 장시간 비활성 상태는 자동 복구 대상이 아니라 재로그인 대상으로 취급한다.
5. 로그인 실패가 늘면 쿠키보다 먼저 저장소, 배포, 프록시, 도메인 설정을 본다.

## 운영 체크리스트

1. 배포 환경에서 `THREADS_WEB_STORE_BACKEND=postgres`와 Postgres 연결 문자열이 설정되어 있는지 확인한다.
2. reverse proxy가 `Host` 헤더를 올바르게 전달하는지 확인한다.
3. `Secure` 쿠키가 HTTPS 구간에서만 사용되는지 확인한다.
4. `/api/public/bot/session`과 `/api/public/bot/plus` 응답의 401 비율을 모니터링한다.
5. Postgres 백업과 복구 절차를 정기적으로 검증한다.
6. 배포 직후 로그인 유지 여부를 확인한다.

## 실무 규칙

- 로그인 풀림이 발생하면 먼저 세션 쿠키 문제로 단정하지 않는다.
- 비접속 30일 초과는 정상 만료로 분류한다.
- 세션이 배포 직후 사라지면 파일 저장소가 아니라 운영 DB 설정부터 확인한다.
- admin 세션과 bot 세션은 별도 경계로 본다.

## 수용 기준

- 배포 후 기존 로그인 세션이 유지된다.
- active user는 정상 사용 중 세션 만료를 체감하지 않는다.
- idle 30일 초과는 명시적 재로그인으로만 풀린다.
- 로그인 장애가 발생해도 원인 분류가 저장소, 프록시, 쿠키, 만료 중 하나로 좁혀진다.
