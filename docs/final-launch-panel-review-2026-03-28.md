# 최종 출시 전 전문가 패널 토론 기록

기준일: `2026-03-28`

프로젝트: `Threads Archive / ss-threads`

## 1. 문서 목적

이 문서는 최종 출시 직전 시점에서 제품을 냉정하게 검토하기 위해 진행한 가상 전문가 패널 토론 내용을 정리한 의사결정 메모다.

참여 역할:

- 웹서비스 기획 전문가
- 마케팅 전문가
- 보안 전문가
- 개발 전문가
- 스레드 헤비유저 그룹 대표

검토 기준:

- 사용자 관점에서 무엇을 이해하고 어떻게 시작하는지
- 실제 구매 전환이 가능한지
- 출시 직후 운영 사고 가능성이 있는지
- 코드와 운영 설정이 실제로 일치하는지

## 2. 결론 요약

현재 상태 평가는 다음과 같다.

- `soft launch`: 가능
- `기존 유저 대상 제한 공개`: 가능
- `공개 유료 full launch`: 보류 권고

판단 이유는 제품 방향 자체보다 다음 네 가지가 더 크다.

- 랜딩 페이지의 제품 구조와 CTA 연결이 일부 실제 동작과 어긋남
- 제품 약속이 한 줄로 정리되지 않아 첫 이해 비용이 높음
- 구매 신뢰와 저장 신뢰 UX가 아직 약함
- 운영 설정과 persisted storefront 설정 간 드리프트 가능성이 있음

## 3. 핵심 근거

검토에 사용한 주요 문서와 코드:

- `COMMERCIAL_READINESS_REPORT.md`
- `docs/deployment-architecture.md`
- `docs/business-plan-summary.md`
- `README.md`
- `packages/web-client/src/lib/web-i18n.ts`
- `packages/web-client/src/landing/main.ts`
- `packages/web-client/src/checkout/main.ts`

특히 실제 출시 blocker로 본 항목은 랜딩의 제품 정의와 CTA 링크 매핑 불일치다.

- `packages/web-client/src/lib/web-i18n.ts`의 bot landing copy에서는 `productA`가 mention bot, `productB`가 Chrome extension으로 정의되어 있다.
- 같은 파일의 landing link 정의에서는 `productAHref`가 extension 다운로드 URL, `productBHref`가 `/scrapbook`으로 설정되어 있다.
- `packages/web-client/src/landing/main.ts`에서는 그 링크가 hero CTA와 product card CTA에 그대로 들어간다.

즉, 사용자에게 보여 주는 제품 설명과 실제 이동 경로가 뒤집혀 있을 가능성이 있다.

## 4. 패널별 토론 내용

### 4-1. 웹서비스 기획 전문가

첫 인상에서 가장 큰 문제는 기능 수가 아니라 진입 경로 설계다.

- 랜딩에서 extension, mention scrapbook, Plus, watchlists, insights가 동시에 보이기 때문에 사용자가 자기 상황에 맞는 시작 경로를 곧바로 고르기 어렵다.
- 특히 `모바일 저장`과 `PC 저장`이 서로 다른 방식인데도 한 제품 설명 아래에서 동시에 소화되다 보니 이해 비용이 올라간다.
- 현재 랜딩은 "두 제품"을 나란히 소개하는 구조지만, 실제로는 "Threads 저장"이라는 하나의 목적 아래 진입 수단이 둘인 형태로 재정리하는 편이 더 맞다.

기획 관점 결론:

- 랜딩은 `한 제품 / 한 약속 / 두 진입 경로` 구조로 다시 묶어야 한다.
- CTA 라우팅 오류는 카피 문제가 아니라 출시 blocker다.

### 4-2. 마케팅 전문가

오퍼 자체는 존재하지만 기억될 정도로 압축되어 있지 않다.

- 현재 메시지는 정확하지만 길다.
- 사용자는 "무엇을 사는가"보다 "왜 지금 이걸 써야 하는가"를 먼저 이해해야 한다.
- Plus 설명은 기능 묶음처럼 보이지만, 핵심 구매 이유 하나가 전면에 나오지 않는다.

구매 신뢰 측면의 문제도 지적되었다.

- 체크아웃 카피에 이미 `payment confirmation 후 키 발송`, `orders are reviewed first`가 들어 있어 즉시 활성화 SaaS처럼 느껴지지 않는다.
- 솔직한 설명 자체는 맞지만, 기대치를 정교하게 맞추지 않으면 전환률이 떨어질 가능성이 높다.

마케팅 관점 결론:

- 제품 약속을 한 문장으로 다시 정의해야 한다.
- 랜딩의 1차 CTA는 구매보다 `무료로 시작` 또는 `설치/저장 시작` 중심이 더 자연스럽다.
- 수동 확인이 섞인 결제라면 SLA를 전면에 드러내야 한다.

### 4-3. 보안 전문가

치명적 P0 blocker는 없지만, 출시 전 최소 보안 게이트는 아직 남아 있다.

- `POST /api/public/orders`에 IP 기반 rate limit이 필요하다.
- 공개 landing/checkout 페이지에 보안 헤더를 더 적용해야 한다.
- reverse proxy 뒤 운영 시 `THREADS_WEB_TRUST_PROXY_ALLOWLIST` 설정이 빠지면 IP 기반 보호가 기대대로 동작하지 않을 수 있다.

보안 관점 결론:

- 대형 취약점보다는 운영 규율 미준수에 따른 보호 장치 무력화가 더 현실적인 위험이다.
- 출시 전 점검 항목으로 강제하는 편이 맞다.

### 4-4. 개발 전문가

개발 관점에서 가장 현실적인 리스크는 코드 결함보다 운영 드리프트다.

- storefront 설정은 DB persisted 값이 우선이므로, 가격/플랜/FAQ 문구를 코드에서 바꿔도 운영 페이지가 바로 바뀌지 않을 수 있다.
- 실제 릴리스에서는 코드 배포와 `/api/admin/storefront-settings` 반영이 함께 움직여야 한다.
- 즉, 현재는 소스 오브 트루스가 완전히 하나가 아니다.

개발 관점 결론:

- 출시 전 체크리스트에 persisted storefront 동기화를 명시적 게이트로 넣어야 한다.
- 랜딩 CTA 매핑 오류는 코드 수정 후 바로 검증이 필요한 실제 버그다.

### 4-5. 스레드 헤비유저 그룹 대표

mention scrapbook 방식은 매력은 있지만 대중적이진 않다.

- 공개 댓글에 `@bot` 멘션으로 저장 요청을 남기는 구조는 익숙해지면 빠르지만, 처음 접한 사용자는 규칙을 배워야 한다.
- 저장 반영이 수집 주기 기반으로 이뤄질 경우 사용자는 "내 요청이 들어갔는지"를 즉시 알고 싶어 한다.
- 불편의 본질은 지연 그 자체보다 저장 신뢰의 부족이다.

헤비유저 관점 결론:

- `저장 대기`, `마지막 수집 시각`, `예상 반영 시간`, `실패 이유`, `재시도 방법`을 더 명확히 보여 줘야 한다.
- 모바일 mention 저장은 핵심 가치가 될 수 있지만, 일반 사용자를 위한 안내 밀도와 상태 피드백은 더 필요하다.

## 5. 패널 합의 사항

### 5-1. 출시 전 필수 수정

1. 랜딩의 `productA / productB` 설명과 CTA 링크 매핑을 일치시킨다.
2. `POST /api/public/orders`에 IP rate limit을 넣는다.
3. landing/checkout 공개 페이지 보안 헤더를 적용한다.
4. reverse proxy 운영 환경에서 `THREADS_WEB_TRUST_PROXY_ALLOWLIST`를 포함한 실제 배포 설정을 검증한다.
5. persisted storefront 설정과 코드 기본값의 불일치를 배포 체크리스트에서 강제로 점검한다.

### 5-2. 출시 전 권고 수정

1. 랜딩 메시지를 `Threads 저장` 하나로 압축하고, 진입 경로만 `PC extension`과 `모바일 mention scrapbook`으로 분리한다.
2. 체크아웃에서 `즉시 활성화`인지 `결제 확인 후 발급`인지 기대치를 더 분명히 맞춘다.
3. Plus 가치 제안을 기능 나열보다 사용 이유 중심으로 다시 정리한다.
4. mention scrapbook에 상태 피드백을 더 추가한다.

### 5-3. 출시 후 2주 집중 관찰 항목

- 저장 반영 지연 시간
- 저장 실패율
- 무료 한도 도달률
- Plus 전환률
- storefront drift 발생 사례
- 주문 생성 abuse 시도

## 6. 최종 판정

한 줄 결론은 다음과 같다.

`제품 방향은 맞다. 하지만 지금 바로 크게 밀면 제품 완성도보다 랜딩 구조 오류, 구매 신뢰 문제, 운영 드리프트가 먼저 발목을 잡을 가능성이 높다.`

출시 판단은 다음과 같이 정리한다.

- 제한적 소프트 런치: `진행 가능`
- 일반 공개 유료 런치: `우선 수정 후 재판정`
